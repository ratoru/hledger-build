# How to use `hledger-build`

`hledger-build` converts raw bank CSV exports into hledger journal files and
generates standard financial reports — automatically, incrementally, and
reproducibly. Every output is cached by a content hash; re-running only rebuilds
what actually changed.

---

## Quick start

```bash
# Install hledger first: https://hledger.org/install.html
hledger-build init    # scaffold a new project
hledger-build         # run the full pipeline
```

`init` creates a working example with sample data so you can see the pipeline in
action immediately. Replace the sample files with your own statements and rules
when you're ready.

---

## Directory layout

After `hledger-build init`, your project looks like this:

```
my-ledger/
├── hledger-build.toml          # config (everything is optional / has defaults)
├── commodities.journal         # commodity format declarations (shared by all years)
├── all.journal                 # top-level file for ad-hoc multi-year queries
├── 2026.journal                # per-year entry point — YOU maintain this file
│
├── sources/
│   ├── 2026-imports.journal    # auto-generated: include lines for all source journals
│   ├── _manual_/               # hand-written journal entries (not imported)
│   │   └── 2026/
│   │       └── opening.journal
│   │
│   └── mybank/                 # one directory per data source
│       └── checking/           # optional subdir
│           ├── mybank.rules    # hledger CSV import rules
│           ├── preprocess      # optional: shell script to clean raw CSV
│           ├── raw/            # ← drop your bank statement exports here
│           │   └── 2026/
│           │       └── transactions.csv
│           ├── cleaned/        # generated by preprocess (if present)
│           │   └── 2026/
│           │       └── transactions.csv
│           └── journal/        # generated by hledger import
│               └── 2026/
│                   └── transactions.journal
│
├── reports/                    # all generated reports land here
│   ├── 2026-all.journal
│   ├── 2026-accounts.txt
│   ├── 2026-income-expenses.txt
│   ├── 2026-balance-sheet.txt
│   ├── 2026-cashflow.txt
│   ├── 2026-unknown.journal
│   └── 2026-metrics.txt
│
└── .build/                     # content-hash cache (add to .gitignore)
    └── manifest.json
```

Key conventions:

- **Sources are auto-discovered**: any directory under `sources/` that contains
  a `raw/` subdirectory is treated as a data source. `_manual_` is excluded.
- **Years are auto-discovered**: 4-digit subdirectory names under any source's
  `raw/` directory, under `sources/_manual_/`, or under `prices/`.
- **Rules files are discovered breadth-first**: `*.rules` files are searched
  from `sources/` down to the source directory, broadest first.

## The ingest pipeline

For each data source, hledger-build runs two stages:

```
raw/{year}/*.csv  →[preprocess]→  cleaned/{year}/*.csv  →[convert]→  journal/{year}/*.journal
```

### The `preprocess` script

If a file named `preprocess` exists in the source directory, hledger-build runs
it before the CSV import. It receives the raw CSV path as `$1` and must write
the cleaned CSV to stdout:

```sh
#!/usr/bin/env sh
# Strip the first 5 lines (bank's summary header) before the real CSV starts
set -eu
tail -n +6 "$1"
```

The `preprocess` stage is **optional**. If the script is absent, the convert
stage reads directly from `raw/` instead of `cleaned/`.

### The convert stage — hledger or a custom script

By default, the convert stage calls `hledger` with your `.rules` file to import
the CSV. This is the right choice for most bank exports.

For sources where hledger's CSV import doesn't fit — a proprietary file format,
a script that generates journal entries from price data, or any complex
transformation — you can place a script named `convert` (or any name you
configure) in the source directory. Like `preprocess`, it receives the input
file path as `$1`, runs from the source directory as the working directory, and
writes journal content to stdout:

```sh
#!/usr/bin/env sh
# convert — generate a journal from a brokerage report
set -eu
python3 parse_brokerage.py "$1"
```

The key difference from `preprocess`: this script's output IS the final
`.journal` file, so it must produce valid hledger journal syntax.

### The `.rules` file (hledger convert only)

hledger uses a `.rules` file to map CSV columns to journal fields and classify
transactions. Rules files are discovered automatically from `sources/` down to
your source directory, broadest first — so you can share common rules at a
higher level and override them per source.

A minimal rules file:

```
# Skip the CSV header row
skip 1

# Map columns by position
fields date, description, amount

# Date format
date-format %Y-%m-%d

# The account this data source represents
account1 assets:mybank:checking

# Default counterpart (transactions that match no rule go here)
account2 expenses:unknown

# Classification rules — top-to-bottom; later rules override earlier ones
if PAYROLL
  account2 income:salary

if GROCERY|SUPERMARKET
  account2 expenses:food:groceries
```

See the [hledger CSV docs](https://hledger.org/hledger.html#csv-format) for the
full rules reference.

## Year journals

hledger-build needs a `{year}.journal` file at the project root for each year.
**You create and maintain these files.** `hledger-build init` creates one for
the current year; you must create new ones manually when you add a new year.

The year journal is the entry point for report generation. **You do not need to
list every source journal manually** — hledger-build auto-generates
`reports/{year}-imports.journal` before each run, listing `include` directives
for all source journals produced by the ingest pipeline for that year. Your year
journal only needs to reference the stub and your hand-written entries:

```ledger
; 2026.journal
include commodities.journal
include sources/_manual_/2026/opening.journal
include sources/2026-imports.journal

; For years after the first, also include the auto-generated opening balances:
; include reports/2026-opening.journal
```

**When you add a new source**, hledger-build automatically adds it to the
imports stub on the next run — no changes to your year journals needed.

**When you add a new year**, create `{year}.journal` with the same three-line
pattern above, adjusting the year number.

`all.journal` is a convenience file for running ad-hoc multi-year queries:

```ledger
; all.journal
include 2026.journal
include 2025.journal
```

### Commodity declarations

`commodities.journal` declares the canonical display format for each commodity
you use. hledger normally infers the format from the amounts it encounters, but
explicit declarations guarantee consistent formatting regardless of which
transactions are seen first:

```ledger
; commodities.journal
commodity $1,000.00
commodity £1,000.00
commodity AAPL 1.0000
```

Every year journal includes `commodities.journal` at the top. When you add a
new commodity or asset, add its directive here once and all years pick it up
automatically.

### Opening balances

The `sources/_manual_/{year}/opening.journal` file records your real account
balances at the start of tracking. Use balance assertions (`=`) to let hledger
verify the running totals:

```ledger
2026-01-01 opening balances
    assets:mybank:checking         = 5000.00
    assets:cash                    =  200.00
    liabilities:credit-card        =  -85.00
    equity:opening balances
```

For years after the first year, hledger-build automatically generates
`reports/{year}-closing.journal` and `reports/{year}-opening.journal` from the
previous year's equity. Replace the hand-written `opening.journal` include with
`include reports/{year}-opening.journal` to carry forward balances automatically.

## Generated reports

hledger-build generates these reports for every year in your configured range:

| Output file                  | Content                                      |
| ---------------------------- | -------------------------------------------- |
| `{year}-all.journal`         | All transactions (`hledger print`)           |
| `{year}-accounts.txt`        | Account list (`hledger accounts`)            |
| `{year}-income-expenses.txt` | Income statement                             |
| `{year}-balance-sheet.txt`   | Balance sheet                                |
| `{year}-cashflow.txt`        | Cash flow statement                          |
| `{year}-unknown.journal`     | Unclassified transactions for review         |
| `{year}-metrics.txt`         | Monthly personal finance metrics (see below) |

For multi-year setups, hledger-build also generates:

- `reports/{year}-closing.journal` — closes income/expense accounts at year-end
- `reports/{year}-opening.journal` — opens asset/liability balances for the next year

### Financial metrics report

The metrics report computes personal finance numbers month by month:

```
Financial Metrics: 2026
Generated: 2026-03-02

Month         Daily Exp    Daily Inc    Net Worth   Savings%  Short Run   Long Run        FIRE#
---------- ------------ ------------ ------------ ---------- ---------- ---------- ------------
   2026-01         6.45       103.23      4200.00      93.8%        31d       651d     58870.97
```

| Metric    | Meaning                                                           |
| --------- | ----------------------------------------------------------------- |
| Daily Exp | Average daily spending (excluding configured accounts)            |
| Daily Inc | Average daily take-home income (excluding configured accounts)    |
| Net Worth | Assets minus liabilities at end of period                         |
| Savings%  | `(take-home − expenses) / take-home`                              |
| Short Run | Liquid cash ÷ daily expense                                       |
| Long Run  | Total assets ÷ daily expense                                      |
| FIRE#     | Annual expenses × FIRE multiplier (default 25 = 4% rule)          |
| AAW / PAW | Average/Prodigious Accumulator of Wealth targets (requires `age`) |

## `hledger-build.toml` reference

Everything is optional.

```toml
# ── Year range (auto-discovered from raw/{year}/ dirs) ───────────────────────
# first_year   = 2014
# current_year = 2026

# ── Global settings ───────────────────────────────────────────────────────────
# hledger_binary = "hledger"
# equity_query   = "assets|liabilities|debts"  # used for closing/opening journals
# jobs           = 0                           # 0 = runtime.NumCPU()

# ── Directory name overrides ──────────────────────────────────────────────────
# [directories]
# sources  = "sources"
# reports  = "reports"
# raw      = "raw"
# cleaned  = "cleaned"
# journal  = "journal"
# build    = ".build"
# prices   = "prices"
# manual   = "_manual_"

# ── Built-in reports ──────────────────────────────────────────────────────────
# All enabled by default. Add a section only to change args or disable a report.
#
# [reports.transactions]
# args    = ["print"]
# enabled = true
#
# [reports.accounts]
# args    = ["accounts"]
# enabled = true
#
# [reports.income_expenses]
# args    = ["is", "--flat", "--no-elide", "--cost"]
# enabled = true
#
# [reports.balance_sheet]
# args    = ["balancesheet", "--no-elide"]
# enabled = true
#
# [reports.cashflow]
# args    = ["cashflow", "not:desc:(opening balances)", "--no-elide"]
# enabled = true
#
# [reports.unknown]
# args    = ["print", "unknown"]
# enabled = true

# ── Metrics report ───
# [reports.metrics]
# enabled     = true
# fire_factor = 25    # FIRE multiplier (4% rule = 25)
# age         = 0     # set to your age to enable AAW/PAW columns (0 = skip)
# currency  = ""      # e.g. "USD" to convert all values via --value=end

# [reports.metrics.accounts]
# exclude_expenses = ["expenses:gross"]   # payroll deductions, excluded from daily avg
# exclude_income   = ["income:gift"]      # windfalls, excluded from daily avg
# cash_assets      = "assets:cash"        # liquid cash account for short runway
```

### Custom reports

Custom reports are scripts invoked after the ingest pipeline completes. Their
stdout is captured to the output file.

**One report per year** — use `years = "all"` with `{year}` in `output` or `args`:

```toml
[[custom_reports]]
name   = "investments"
output = "{year}-investments.txt"
script = "./investments.sh"
years  = "all"
args   = ["{year}"]
```

**One report across all years** — `years = "all"` without `{year}` in `output`:

```toml
[[custom_reports]]
name   = "net-worth-history"
output = "net-worth-history.txt"
script = "./net-worth.sh"
years  = "all"
```

**Year-range report** — use `year_range`. The value `"current"` resolves to the
current year. Use `{from_year}` and `{to_year}` in `output` and `args`:

```toml
[[custom_reports]]
name       = "rolling-5-year"
output     = "{from_year}-{to_year}-summary.txt"
script     = "./rolling.sh"
year_range = {from_year = "2022", to_year = "current"}
```

**Explicit dependencies** — for reports that depend on specific generated files:

```toml
[[custom_reports]]
name       = "combined"
output     = "combined.txt"
script     = "./combine.sh"
depends_on = ["2026-all.journal", "2025-all.journal"]
```

Entries in `depends_on` without a path separator are resolved relative to the
reports directory.

**Generated journals (`dyngen`)** — when a custom report script produces a
`.journal` file that is itself `include`d by a year journal, set `dyngen = true`.
This prevents a circular dependency (the step would otherwise depend on its own
output):

```toml
[[custom_reports]]
name   = "mortgage"
output = "{year}-mortgage-interest.journal"
script = "./mortgage.sh"
years  = "all"
dyngen = true   # output is included by {year}.journal — break the circular dep
```

### Per-source pipeline overrides

**Extra pipeline stage** — splice an additional stage into the default pipeline:

```toml
[[sources."mybank/checking".extra_stages]]
name       = "dedupe"
script     = "./dedupe.sh"
after      = "preprocess"    # insert after this stage name
input_dir  = "cleaned"
input_glob = "*.csv"
output_dir = "deduped"
output_ext = ".csv"
```

**Full pipeline replacement** — replace the entire pipeline for one source:

```toml
[[sources."mybroker/brokerage".pipeline]]
name       = "preprocess"
script     = "./preprocess"
input_dir  = "raw"
input_glob = "*.csv"
output_dir = "cleaned"
output_ext = ".csv"
optional   = true

[[sources."mybroker/brokerage".pipeline]]
name       = "convert"
script     = "./convert"     # custom script instead of hledger
input_dir  = "cleaned"
input_glob = "*.csv"
output_dir = "journal"
output_ext = ".journal"
```

**Extra dependencies** — declare additional files a source depends on (e.g.
price files used by a custom convert script):

```toml
[sources."mybroker/brokerage"]
extra_deps = ["prices/2026/prices.csv"]
```

## Typical workflow

1. **Export** your bank statement as CSV and drop it into
   `sources/{source}/raw/{year}/`.
2. **Run** `hledger-build`. The ingest pipeline imports the CSV; reports are
   regenerated automatically.
3. **Review** `reports/{year}-unknown.journal` for unclassified transactions.
4. **Categorize** unknowns: run `hledger-build categorize` (interactive fzf
   workflow), or manually add rules to your `.rules` file.
5. **Run again** to apply the new rules.
6. **Commit** everything — raw exports, rules, journals, reports — to version
   control.

## FAQ and Tips

### Record your gross income

Accurate gross income helps with savings rate and tax calculations. Bank
statements show only the net deposit. If you receive a salary, add a monthly
adjustment in `_manual_`:

```ledger
; [Auto-generated from bank statement]
2026-03-01 Bank Deposit
  assets:checking     $2800.00
  income:salary      $-2800.00

; [Added manually from payslip]
2026-03-01 Payslip Adjustment
  expenses:taxes                 $400.00
  expenses:gross:retirement      $200.00
  income:salary                 $-600.00
```

The metrics report excludes `expenses:gross` by default when computing daily
spending, so payroll deductions don't inflate your expense averages.

### How do you balance transfers between 2 accounts when you have statements for both accounts? {#transfer-2-accounts}

#### The Problem

In your primary bank account you've been classifying transfers to a secondary
account as `expenses:OtherAccount`. But you've recently started processing the
statements from the second account as well, and now balances are wrong because
both statements record the same transfer.

In `bank1.journal`, imported from `bank1.csv`:

```
2018/11/09 Transfer
    assets:Bank1:Primary    $-200
    assets:Bank2:Secondary
```

In `bank2.journal`, imported from `bank2.csv`:

```
2018/11/09 Transfer
    assets:Bank2:Secondary  $200
    assets:Bank1:Primary
```

#### The Solution

Introduce an intermediate account such as `assets:Transfers:*`. The transfer
moves from one account to the intermediate, and then from the intermediate to
the other account:

In `bank1.journal`:

```
2019-05-18 Transfer
    assets:Bank1:Primary         $-200
    assets:Transfers:Bank1Bank2
```

In `bank2.journal`:

```
2019-05-18 Transfer
    assets:Bank2:Secondary       $200
    assets:Transfers:Bank1Bank2
```

The balance of `assets:Transfers` should always be zero. When it isn't, you're
missing one side of a transfer or a rule is wrong.

I typically sort the account names alphabetically for the subaccount name:

```python
"assets:Transfers:" + "".join(sorted(["Bank2", "Bank1"]))
```

#### Broadly applicable

This pattern applies whenever you add a new data source that overlaps with an
existing one:

- **PayPal**: change bank rules to send PayPal transactions to
  `assets:PayPal:top-up`; set PayPal's money-in to `assets:PayPal:top-up` as
  well.
- **Credit card**: bank records the payment as a liability transfer; credit card
  statements record the individual charges.
- **Amazon, crypto exchanges, debit card providers**: same idea — use an
  intermediate account to avoid double-counting.

This approach is described in detail in
[Full-fledged hledger](https://github.com/adept/full-fledged-hledger/wiki/Adding-more-accounts#lets-make-sure-that-transfers-are-not-double-counted).
